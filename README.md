# Driver Drowsiness Detection System

A real-time computer vision system that detects driver fatigue from a live webcam feed and triggers an audible alarm before a lapse in attention becomes dangerous.

## Overview

The system watches the driver's face through a webcam, tracks the eyes using 468 facial landmarks, and combines two independent signals — a geometric measure (Eye Aspect Ratio) and a learned visual classifier (CNN) — to decide whether the eyes are open or closed. A consecutive-frame counter filters out normal blinking so the alarm only fires on sustained eye closure, which is a strong indicator of drowsiness.

## Tech Stack

- **Python 3.13**
- **OpenCV** — video capture, image processing, drawing overlays
- **MediaPipe (Face Landmarker)** — real-time 468-point facial landmark detection
- **TensorFlow / Keras** — CNN for binary eye-state classification (Awake / Sleepy)
- **Pygame** — alarm playback
- **NumPy**

## How It Works

**1. Face landmark detection.** Each webcam frame is passed to MediaPipe's Face Landmarker, which returns 468 normalized (x, y, z) points describing the face. Unlike Haar Cascades, this landmark mesh keeps tracking the eye region even when the eyes are fully closed, since it models the whole face rather than searching for an "open eye" pattern.

**2. Eye Aspect Ratio (EAR).** For each eye, 6 landmark points are used to compute:

```
EAR = (‖p2 - p6‖ + ‖p3 - p5‖) / (2 × ‖p1 - p4‖)
```

EAR stays roughly constant while the eye is open and drops sharply when it closes. This gives a fast, lighting-independent signal with almost no computational cost.

**3. CNN classification.** The eye region (located via the same landmarks) is cropped, converted to grayscale, resized to 64×64, and passed through a 3-block convolutional neural network trained on the MRL Eye Dataset (~68,000 labeled eye images) to independently predict Awake / Sleepy.

**4. Combined decision.** A frame is only flagged "sleepy" when **both** EAR and the CNN agree. Requiring agreement between a geometric signal and a learned signal reduces false positives that either method would produce alone (e.g. EAR dipping briefly during a normal blink, or the CNN misfiring on a single noisy frame).

**5. Consecutive-frame counter.** Drowsiness is only declared after a set number of consecutive "sleepy" frames (not a single frame), which distinguishes a natural blink (a few frames) from sustained eye closure (many consecutive frames). This is what keeps the alarm from firing on every blink.

**6. Alert.** Once the threshold is crossed, an on-screen "DROWSINESS ALERT!" warning appears and a looping alarm sound plays via Pygame until the eyes reopen.

## Setup & Usage

### 1. Install dependencies
```bash
python -m venv venv
venv\Scripts\activate          # Windows
pip install -r requirements.txt
```

### 2. Download the MediaPipe Face Landmarker model
```
https://storage.googleapis.com/mediapipe-models/face_landmarker/face_landmarker/float16/latest/face_landmarker.task
```
Place it in the project root as `face_landmarker.task`.

### 3. Get the dataset
This project uses the **MRL Infrared Eye Images Dataset** (Awake/Sleepy split), available on Kaggle. Update the `TRAIN_DIR` / `VALID_DIR` paths in `train_model.py` to point to the extracted dataset.

### 4. Train the model
```bash
python train_model.py
```
Saves the trained model to `models/eye_cnn_model.h5`.

### 5. Add an alarm sound
Place any short `.mp3` alarm/beep clip in the project root as `alarm.mp3`.

### 6. Run real-time detection
```bash
python detect.py
```
Press `q` to quit.

## Tuning

- `EAR_THRESHOLD` in `detect.py` (default `0.21`) — lower it if the alarm triggers too easily, raise it if closed eyes aren't being caught.
- `DROWSY_FRAME_THRESHOLD` (default `15`) — number of consecutive sleepy frames required before the alarm fires. Higher = more tolerant of long blinks; lower = faster response.

## Possible Extensions

- Yawn detection using mouth landmarks
- Head-pose estimation to catch head-nodding as an additional fatigue signal
- Logging drowsiness events with timestamps for a driver behavior report
- Deploying on a Raspberry Pi with an in-cabin camera for a real embedded prototype
